# Recommendation Rubric Reference

## Core Principle

The recommendation is synthesis of pre-computed reviewer verdicts and analysis numbers. Every claim cites a specific number from the plan or a verdict payload. "Looks promising" is not a number.

## Required Sections

Emit exactly four sections, in this order.

1. **Batch summary** — N experiments, breakdown by status code with integer counts per vocabulary term, total wall-clock seconds for the batch.
2. **Verdicts** — one line per experiment carrying the experiment ID, the primary metric value, the metric reviewer verdict, the strategy reviewer verdict, and the recommended action from the set winner-candidate, discard, needs-human-attention, blocked.
3. **Cross-batch trend** — best metric across the plan so far, gap between that best and the success criteria from the hypothesis, diminishing-returns assessment pulled from the analysis. Omit this section entirely on the first batch.
4. **Suggested next direction** — exactly one value from the four allowed strings defined below, backed by the specific numbers that triggered it.

## Inline by Controller

The recommendation is generated by the report controller, not by a subagent. Reviewers produce verdicts that demand structural distrust; the recommendation is synthesis of those verdicts plus analysis numbers, never raw metric interpretation by the author of the recommendation.

## Cited Numbers

Every quantitative claim cites its source. Acceptable source tags are the plan field name, the reviewer payload key, or the analysis output key. Vague comparisons are forbidden. A verdict line without a numeric metric value and a payload reference is malformed. A trend claim without a cited slope or gap value is malformed.

## Suggested-Next Categories

The suggested-next-direction field carries one and only one of these strings, with the listed trigger condition.

- "approve winner X and run another batch in direction Y" — at least one experiment has KEEP from review-strategy and sits on the Pareto front.
- "reject batch and re-plan via /design-experiments" — every reviewed experiment is REJECT or DISCARD, and the failure pattern points at architecture-level issues.
- "kill plan and re-frame via /hypothesize" — the diminishing-returns curve plateaus across three or more batches AND no experiment has closed the success-criteria gap.
- "all-failure batch, retry with adjusted bounds" — every experiment is BLOCKED_PREFLIGHT, CRASHED_OOM, or BLOCKED_BUILD, pointing at environment or bound issues rather than research issues.

## Gate Functions

- BEFORE writing the summary: "Are status counts derived from the plan's Results sections, or am I estimating?"
- BEFORE writing a verdict line: "Does it cite the metric value and the reviewer payload?"
- BEFORE writing the suggested next direction: "Does the trigger condition match the analysis I just computed?"
- BEFORE printing: "Is every quantitative claim sourced?"

## Rationalization Table

| Rationalization | Response |
|---|---|
| The numbers are obvious from the dashboard, terminal can be brief | The recommendation is the human's read-once channel. Cite numbers explicitly. |
| I can say 'looks promising' if the trend is up | Cite the trend slope. "Looks promising" is not a number. |
| The strategy reviewer said KEEP_WITH_CONCERNS, I'll round it to KEEP | Carry the verdict verbatim. The concern is the point. |
| I'll guess the next direction since the analysis was inconclusive | Inconclusive analysis maps to the "all-failure" or "kill plan" path, not a guess. |

## Red Flags

- "Looks promising" / "seems good" / "appears to converge" anywhere in the recommendation
- Verdict line missing the metric value or reviewer payload reference
- Suggested next direction not in the four allowed categories
- Cross-batch trend section present on the first batch
- Quantitative claim with no cited source

## Bottom Line

Write and execute a script that parses the printed recommendation text and asserts: exactly four sections present in the required order, every verdict line carries a numeric metric value, the suggested next direction string matches one of the four allowed values, every quantitative claim carries a [plan:field] or [reviewer:key] source tag. Print `RECOMMENDATION RUBRIC: PASS` or `RECOMMENDATION RUBRIC: FAIL` with the specific failing rule.
